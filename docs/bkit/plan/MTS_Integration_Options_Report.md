# [의사결정 요청] MTS 알림톡/문자 발송 모듈 독립 연동 방안

현재 Smallbee 시스템에 종속되어 있는 알림톡/문자 발송 모듈을 Jeisys(현 서비스) 환경에 독립적으로 구축하기 위한 아키텍처 연동 방안 보고서입니다. AWS Staging 환경을 고려하여 다음 2가지 연동 방식을 제안하오니, 인프라 담당자 및 개발팀 논의를 거쳐 채택 부탁드립니다.

## 💡 연동 방식 요약 비교

| 구분 | Option A: DB 에이전트 방식 (기존 유지) | Option B: REST API 방식 (모던 클라우드) |
|---|---|---|
| **연동 방식** | AWS Staging용 MySQL에 MTS 테이블을 만들고 MTS 에이전트가 이를 폴링 | Jeisys 백엔드에서 MTS 컴퍼니의 발송 API 엔드포인트를 HTTP로 직접 호출 |
| **개발 공수** | **낮음** (기존 스몰비 로직 재사용 가능) | **보통** (신규 HTTP 발송 모듈 개발 필요) |
| **인프라 공수** | **높음** (서버에 에이전트 설치 및 데몬/포트 관리 필요) | **낮음** (인프라 세팅 없이 코드만으로 해결, Docker 친화적) |
| **DB 환경** | MySQL 필수 (MTS 에이전트 구동용) | 제약 없음 (현재 SQLite/Prisma 환경에서도 즉시 가능) |
| **권장 환경** | 레거시 호환성 유지, 백엔드 로직 수정을 최소화하고 싶을 때 | 유지보수 편의성 및 클라우드 인프라 관리를 간소화하고 싶을 때 |

---

## 📌 Option A: DB 에이전트 방식 (기존 스몰비 로직 최대한 유지)

AWS Staging 서버에 구축할 MySQL 인스턴스를 활용하여, MTS 에이전트 프로그램을 직접 설치하고 연동하는 방식입니다.

### 👍 장점
- 복사해 온 기존 스몰비 로직(`smsSendByDbService.js`, `mysqldb.js` 등)을 거의 수정 없이 그대로 재사용할 수 있습니다. 
- 비즈니스 로직(템플릿 생성 등)과 발송 로직(DB Insert)을 기존 구조와 완전히 동일하게 가져가므로 익숙하고 검증된 안정성을 보장합니다.

### 👎 단점
- 서버나 컨테이너 인프라에 외부 데몬 프로그램(MTS 에이전트)을 띄우고 네트워크/방화벽을 열어 지속적으로 모니터링해야 하는 **운영 부담**이 생깁니다.

### 🛠 필요 작업
1. AWS Staging용 MySQL에 MTS 연동 테이블(`MTS_SMS_MSG`, `MTS_ATALK_MSG` 등) 생성 스크립트 작성.
2. 백엔드 `.env`의 DB 접속 정보를 신규 MySQL 인스턴스 주소로 변경.
3. 인프라 담당자 주도로 Staging 서버 OS에 MTS 에이전트 셋업.

---

## 📌 Option B: REST API 방식 (클라우드 친화적)

DB에 큐(Queue) 데이터를 Insert 하는 과정을 생략하고, 백엔드 코드 레벨에서 HTTP 통신으로 MTS 컴퍼니의 발송 API를 직접 호출하는 방식입니다.

### 👍 장점
- 인프라에 추가적인 에이전트를 설치할 필요가 없으며, Docker 및 AWS ECS/Lambda 환경에서도 제약 없이 동작하므로 **AWS 인프라 관리가 훨씬 수월해집니다.**
- Jeisys 내부 발송 이력(History)을 현재 구성된 Prisma ORM 기반에 깔끔하게 쌓을 수 있습니다.

### 👎 단점
- 기존 DB Insert 로직을 버리고 HTTP 호출 기반의 코드를 새로 작성해야 합니다.
- MTS 컴퍼니 측으로부터 **API 연동 매뉴얼과 인증키(Auth Key)**를 사전에 수급해야 합니다.

### 🛠 필요 작업
1. `mysqldb.js`, `smsSendByDbService.js` 등 기존 DB 통신 코어 모듈 폐기.
2. HTTP 통신으로 알림을 전송하는 모듈(`smsSendByApiService.js`) 신규 작성.
3. 발송 요청을 신규 API 서비스로 포워딩하도록 라우터(`wrapper/index.js`) 전면 수정.
