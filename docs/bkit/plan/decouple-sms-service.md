# PDCA Plan: SMS/알림톡 발송 서비스 독립화 (Decoupling from Smallbee)

## 1. 개요 (Plan)
**문제 의식 (사용자 제안):** 
현재 구성된 방식(Smallbee의 `smallbee-sms` 외부 DB에 직접 큐를 Insert 하거나 API를 호출하는 방식)은 Smallbee 서비스나 인프라에 장애가 발생할 경우, 현 서비스(Jeisys)의 결제/주문 알림 등 핵심 기능까지 연쇄적인 장애(Cascading Failure)를 겪게 되는 치명적인 약점이 있습니다.

**목표:** 
Smallbee의 핵심 템플릿 로직과 인터페이스 소스코드는 재사용하되, 실제 메시지를 발송하고 발송 이력을 관리하는 인프라는 현 서비스 내부에 독자적으로 구축하여 결합도(Coupling)를 낮추고 가용성을 극대화합니다.

## 2. 현행 구조의 문제점 분석
1. **단일 장애점(SPOF) 노출**: Smallbee DB가 다운되거나 방화벽 이슈가 생기면 현 서비스에서 모든 알림 발송이 불가해집니다.
2. **안티 패턴 (Integration Database)**: 두 개의 독립적인 도메인(Smallbee, 현 서비스)이 하나의 데이터베이스를 공유하는 구조는 보안, 스키마 변경, 트랜잭션 관리에 매우 취약합니다.
3. **개인정보 침해 리스크**: 고객의 전화번호, 주문 정보 등 민감한 개인정보가 타 서비스의 데이터베이스에 불필요하게 적재됩니다.

## 3. 독립화 아키텍처 제안 (3 Phase)

### Phase 1: 로컬 DB 전환 (발송 이력 및 큐 분리)
- `smallbee-sms` 외부 DB에 쿼리를 날리던 로직(`smsSendByDbService.js`)을 **현 서비스 자체 DB (Prisma / Supabase)**의 테이블로 대체합니다.
- `util/mysqldb.js` 대신 현 서비스의 ORM을 사용하여 발송 요청을 안전하게 저장합니다.

### Phase 2: 외부 벤더사 API 직접 연동 (Direct API Call)
- Smallbee의 발송 워커를 거치지 않고, 현 서비스에서 알림톡/문자 벤더사(예: 알리고, 비즈엠, NHN Cloud 등)의 API를 직접 호출하는 로직을 신규 작성합니다.
- 동기/비동기 발송: 트래픽에 따라 백그라운드 워커(Supabase Edge Function 또는 메인 서버의 Queue)를 활용해 비동기로 발송합니다.

### Phase 3: 전략 패턴(Strategy Pattern) 적용
- 복사해온 `wrapper/index.js`를 리팩토링하여, 내부적으로 `SenderStrategy` 인터페이스를 둡니다. 
- 이를 통해 비즈니스 로직(템플릿 매핑, 텍스트 치환)은 그대로 쓰면서, 끝단에 붙는 발송기(Smallbee용, Aligo용, Bizm용)만 갈아끼울 수 있는 유연한 구조를 만듭니다.

## 4. 기대 효과
- **독립성 보장**: Smallbee 인프라 상태와 무관하게 100% 독자적인 알림 발송 보장.
- **보안 강화**: 민감한 고객 정보의 외부 유출 방지.
- **코드 재사용성**: 잘 만들어진 템플릿 매핑 로직은 살리면서 인프라 의존성만 제거.

## 5. 넥스트 스텝 (확인 필요 사항)
이 방향(독립화)으로 이행하기 위해 다음 사항들을 결정해야 합니다.

1. **발송 벤더사 선정**: 알림톡 및 문자를 직접 발송할 서드파티 벤더사(알리고, 비즈엠, NHN Cloud, 토스트 등)가 이미 정해져 있나요?
2. **DB 환경**: 발송 이력을 저장할 공간은 현재 Jeisys 백엔드에서 사용 중인 DB(Prisma 연결)에 테이블을 신규 생성하여 처리하면 될까요?
