# PDCA Act: 회원관리 하위 크레딧 및 포인트 이력 관리 메뉴 추가와 조회 기능 구현

## 1. 평가 및 회고 (Act)
- **성공 요인**: 기존의 통계 분석 및 크레딧 이력 구현 방식에 맞추어 `point_transactions` 테이블 조인 쿼리(`users!user_id`)를 일관성 있게 구현할 수 있었습니다.
- **개선점**: Supabase API 특성상 nested join 관계(`users!user_id`)에서 `or` 조건으로 하위 테이블 필드를 다중 검색할 때(예: `user.name.ilike`) 한계가 있어, 검색어 매칭 유저의 ID를 선조회한 뒤 `.in` 조건으로 바인딩하는 최적화 패턴을 크레딧과 포인트 전체 이력 관리 API에 모두 적용했습니다. 이는 대용량 데이터 조회 시 인덱스를 유용하게 활용할 수 있어 성능적으로 안정적입니다.

## 2. 향후 계획 (Next)
- 향후 거래 내역이 매우 많아질 경우를 대비하여 `credit_transactions` 및 `point_transactions` 테이블의 `created_at` 컬럼 인덱스 여부를 모니터링하고 인덱스가 미설정되어 있다면 인덱스를 추가하도록 조치할 예정입니다.
